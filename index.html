OSM Data Model

  • 3 element types
    • nodes: points in space
    • ways: linear and area features
    • relations: relationships between nodes and/or ways
  • common properties: a unique id, a list of key-value pairs (i.e. tags), a version, timestamp, user id, and changeset id

OSM Data Model: nodes

  • Single points in space defined by their latitude, longitude and node id.
  • Used for points of interest and other point-shaped objects like shops, individual trees or traffic lights.
  • Also used to define ways
  • example: (bicycle) charging station

OSM Data Model: ways

  • Ordered list of nodes
  • Used for linear-shaped objects like roads and smaller rivers
  • example: residential road

OSM Data Model: areas

  • Ways are also used for polygon-shaped objects like buildings and landuse areas when the way is closed
  • example: Hogent Campus Mercator, Gebouw J

OSM Data Model: closed ways

  • Note that closed ways occasionally represent loops, such as roundabouts on highways or barrier-like structures, rather than solid areas
  • example: fence

OSM Data Model: relation

  • Describes a relationship between two or more data elements: nodes, ways or other relations
  • The elements of a relation are called members
  • Used for “everything else”, i. e. more complex geometric objects like multipolygons and turn restrictions.
  • example: Only Right Turn Restriction

OSM Data Model: tags

  • Add an explicit meaning to a particular element
  • Any OSM element: #tags >= 0, no upper limit
  • Top-level/primary/feature tags vs additional/secondary tags: Map Features

Top-level, primary or feature tags are those that define the nature of map features and therefore may be the sole tags that exist on a map element. All other tags are additional properties describing map features and can be considered additional or secondary tags

OSM model vs GIS “Simple Features” data model

  • Most Geographic Information Systems (GIS) operate on a different data model than OSM
  • OSM data is frequently used in these systems (e.g. QGIS): crucial to convert between the models
  • Zero-, one-, and two-dimensional objects:
    • Point
    • LineString
    • Polygon
  • see Simple Features Standard

Issues with the OSM data model

  • Varying level of detail
  • ID (in)stability
  • Building a navigation graph
  • ...

    • => Evolution of the OSM data model

  • Depending where you are, OSM will more or less detail: sometimes a single closed way is a forest, other times every individual tree of a park is captured as a node
  • ID (in)stability: when a way is split (for attribution reasons), the id is reused for one part, the other part receives a new id
  • routing graph contains: vertices (nodes) and edges (ways) but too large

Mapmaking: OSM data extracts

  • Planet files: weekly “planet” files are made available, these are full dumps of the OSM database.
  • different formats: xml and pbf
    • xml: human readable
    • pbf: custom binary format, based upon Google’s Protocol Buffer binary encoding format
    • pbf is highly compressed, the weekly dump file is about 80G in size (as opposed to 149G for bz2 compressed XML)
    • There are special tools available to export data from a pbf file
https: //en.wikipedia.org/wiki/Bzip2

Changesets: Change file formats

  • Multiple formats to describe the differences between 2 datasets (e.g. 2 planet pbfs), most common is the osmChange format, it’s human readable xml
  • These osmChange files (extension .osc) are published regularly (minutely, hourly, daily), see here
  • Tooling is available to “apply” the changesets on top of an existing pbf, in order to obtain a new pbf

Community: Automatic Edits

"generally" not allowed, see Code of Conduct

Issues:

  • Data Quality: human mappers are better at understanding the local context
  • Licensing issues
  • Conflicts: with contributions from other mappers
  • Community Trust: automated imports can undermine this community-driven approach and lead to distrust among users
  • Data Quality: Automatic imports can lead to poor data quality if not properly vetted. Human mappers are better at understanding local context and ensuring accurate representation of features.
  • Attribution Issues: OSM requires proper attribution for data sources. Automatically importing data may not comply with licensing requirements, leading to potential legal issues
  • Conflict Resolution: Automatic imports can overwrite existing data or create conflicts with contributions from other mappers. Manual editing allows for careful resolution of such conflicts
  • Community Trust: The OSM community relies heavily on the contributions of individual mappers. Automated imports can undermine this community-driven approach and lead to distrust among users.

What is Overture?

  • Founded in 2022 by Amazon, Meta, Microsoft and TomTom as collaborative map building project
  • To be complementary to OSM, not to replace it
  • Makes use of OSM data, but also other datasets
  • Developer focused:
    • Standardized, well structured and documented schema
    • Extensive validation (QA)
    • Easy to integrate and merge different datasets (conflation) using GERS

Global Entity Reference system: two datasets that refer to the same real-word entity should do that through the same identifiers (ids). This makes it easy to merge and combine these different datasets.

Overture Foundation (1)

  • Steering, General and Contributor members have different rights
  • Under the umbrella of the Linux Foundation
  • Steering, General and Contributor + non-profits
  • annual fee
  • contributing headcount
  • steering committee seat
  • influence priorities

Theme: Transportation Network (4)

Overture: The introduction of a max speed change does not lead to ID churn. The technique to "map" certain attributes to subsections of a certain road is called Linear Referencing (LR).

Theme: Transportation Network (5)

Why is ID stability and consistent sectioning so important?

  • Developer focus: with a more stable Transportation Network, your "own" dataset that refers to the transportation elements can be re-applied more easily, when the Transportation Network evolves over time.
  • The process of merging different map datasets (e.g. Transportation Network + Speed Restrictions) is called map conflation

How to use the Overture data?

  • Data is released on a monthly basis
  • Overture Map Explorer
  • Data can be downloaded from cloud storage: Amazon Web Services (AWS) or Microsoft Azure
  • Default format: GeoParquet
    • Parquet: column oriented, modern alternative to CSV, tailored towards fast querying
    • GeoParquet: support for storing geometry types
